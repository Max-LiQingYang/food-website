### 1. AI 食谱生成

- **价值**：高
- **可行性**：高
- **优先级**：P0
- **目标用户**：
  - 家庭主妇/主夫：每天纠结"今天吃什么"
  - 厨房新手：不知道如何处理手头的食材
  - 美食爱好者：想尝试新菜式但缺乏灵感
- **用户故事**：
  > 小张下班回到家，冰箱里只有鸡蛋、番茄和剩米饭。他打开 App，输入"鸡蛋、番茄、米饭"，选择"快手菜"、"中餐"，AI 在 3 秒内生成了一份"番茄蛋炒饭"的完整食谱，包括食材用量、分步做法和预计 15 分钟的烹饪时间。
- **核心功能描述**：
  1. 用户输入：食材清单（多选/手动输入）、口味偏好（清淡/麻辣/酸甜等）、菜系（中餐/西餐/日料等）、烹饪时间限制、难度要求
  2. AI 生成：食谱名称、食材配比（带克重）、分步做法（每步带预计时间）、烹饪总时长、难度等级（1-5 星）、标签（快手菜/下饭菜/减脂餐等）
  3. 用户操作：一键收藏、一键发布（填充到食谱创建表单）、重新生成、微调（"再辣一点"、"换成鸡胸肉"）
  4. 结果保存：生成的食谱可存入用户草稿箱或直接发布到平台
- **技术依赖**：
  - LLM API（DeepSeek / 通义千问 / GPT-4）
  - Prompt 工程 + 结构化输出（JSON Schema）
  - 现有食谱 CRUD 接口复用
- **预估工作量**：2-3 周（1 名后端 + 1 名前端）
  - Prompt 设计与调优：3 天
  - 后端 API 开发（生成 + 保存）：5 天
  - 前端交互页面：5 天
  - 测试与优化：3 天
- **风险/注意事项**：
  - ⚠️ 生成内容的准确性：AI 可能在食材比例、烹饪时间上给出不合理建议，需设置"免责声明"并允许用户反馈修正
  - ⚠️ 内容安全：需过滤危险食谱（如未煮熟的食材搭配）
  - ⚠️ 版权问题：AI 生成的食谱与已有食谱高度相似时的处理策略
  - ✅ 建议：生成结果标注"AI 生成，请根据实际情况调整"，并建立用户举报/反馈机制

### 2. 智能菜单规划

- **价值**：高
- **可行性**：中高
- **优先级**：P1
- **目标用户**：
  - 上班族：想提前规划一周饮食，减少决策疲劳
  - 健身人群：需要按热量/营养素目标安排饮食
  - 家庭用户：需要照顾多人不同口味
- **用户故事**：
  > 李女士一家三口，孩子 8 岁，丈夫正在减脂。她输入"3 人、7 天、孩子不吃辣、丈夫低脂高蛋白"，AI 生成了一周早中晚三餐的规划，每天不重样，还附带了购物清单，告诉她需要买多少鸡胸肉、西兰花和鸡蛋。
- **核心功能描述**：
  1. 用户输入：用餐人数、规划天数（1-7 天）、每餐需求（早餐/午餐/晚餐勾选）、饮食限制（素食/清真/过敏源/低盐低脂等）、预算范围
  2. AI 生成：每日三餐搭配（每餐对应一个具体食谱）、一周膳食概览（总热量、营养均衡度评分）、购物清单（按食材分类汇总，带建议购买量）
  3. 高级功能：食材复用优化（避免买多种只用一次的食材）、剩菜利用建议、营养目标追踪（如每日蛋白质 100g）
  4. 执行支持：每日推送"今日菜单"、一键跳转食谱详情、打卡记录实际执行情况
- **技术依赖**：
  - LLM API（复用 AI 食谱生成的能力）
  - 用户画像/偏好存储
  - 推送服务（可选，初期可用 App 内提醒）
- **预估工作量**：3-4 周（1 名后端 + 1 名前端）
  - 规划算法与 Prompt 设计：5 天
  - 后端 API 开发：7 天
  - 前端页面（日历视图 + 购物清单）：7 天
  - 测试与迭代：4 天
- **风险/注意事项**：
  - ⚠️ 用户实际执行率低：规划很好但用户可能不按计划执行，需设计激励机制（打卡、成就系统）
  - ⚠️ 食材复用的算法复杂度：简单的 LLM 生成可能忽略食材复用，需要在 Prompt 中明确约束或后处理优化
  - ⚠️ 过敏源/饮食禁忌的准确性：涉及健康安全，必须高亮提示"请自行确认食材适用性"
  - ✅ 建议：V1 版本先做基础规划，V2 再加入营养目标追踪和购物清单优化

### 3. 营养分析

- **价值**：中高
- **可行性**：中
- **优先级**：P1（P0 之后的第二批）
- **目标用户**：
  - 健身/减脂人群：精确控制每日摄入
  - 慢病管理人群（糖尿病/高血压）：关注特定营养素
  - 育儿家庭：关注孩子营养均衡
- **用户故事**：
  > 王先生的食谱"红烧肉"获得了 500 个收藏，有用户在评论区问"这道菜热量多少？"王先生自己也不知道。有了营养分析功能后，系统自动为他的食谱标注了"约 450 kcal/份，蛋白质 25g，脂肪 30g，碳水 15g"，还生成了营养标签"高蛋白、高脂肪"。
- **核心功能描述**：
  1. 分析对象：已有食谱（基于食材清单）、用户自定义食材清单、单道菜品
  2. 输出内容：每份热量（kcal）、三大营养素（蛋白质/脂肪/碳水化合物，单位 g）、微量元素（钠、膳食纤维等，可选）、营养标签（如"低脂"、"高蛋白"、"高钠"）
  3. 展示形式：食谱详情页展示营养卡片、营养成分饼图、与每日推荐摄入量的对比条
  4. 批量分析：用户上传一段食材清单文本，AI 解析并计算总营养
- **技术依赖**：
  - 方案 A（推荐）：LLM + 营养数据库混合
    - 中国食物成分表 / USDA FoodData Central（免费）
    - LLM 负责食材识别和克重估算，数据库提供准确营养值
  - 方案 B（快速 MVP）：纯 LLM 估算
    - 实现更快，但准确性较低，适合初期验证需求
- **预估工作量**：3-4 周（1 名后端 + 1 名前端）
  - 营养数据源调研与接入：3 天
  - 食材解析与匹配算法：7 天
  - 后端 API 开发：5 天
  - 前端营养展示组件：5 天
  - 测试与校准（准确性是关键）：5 天
- **风险/注意事项**：
  - ⚠️ **准确性是生死线**：营养数据与健康直接相关，估算误差可能引发用户信任危机甚至健康风险
  - ⚠️ 计量单位歧义："一把葱"、"适量盐"如何量化？需要建立合理的估算规则并在界面明确标注"估算值"
  - ⚠️ 烹饪方式影响：油炸 vs 清蒸，营养值差异巨大，AI 需要结合烹饪方式调整
  - ⚠️ 数据来源：中国食物成分表数据较旧，USDA 是西方食材为主，部分中国特色食材可能缺失
  - ✅ 建议：V1 用 LLM 估算快速上线，标注"仅供参考"；V2 接入专业数据库提升准确性；V3 引入用户校正反馈机制

### 4. 图片识别食材

- **价值**：中
- **可行性**：中
- **优先级**：P2
- **目标用户**：
  - 年轻用户：喜欢拍照分享，对"拍一拍就知道做什么"有好奇心
  - 厨房小白：不认识某些食材，或不知道手头的食材叫什么名字
  - 买菜场景：在菜市场看到新奇食材，拍照识别后获取做法
- **用户故事**：
  > 小陈在菜市场看到一种不认识的绿色蔬菜，拍了一张照片上传到 App，AI 识别出是"秋葵"，并推荐了"白灼秋葵"、"秋葵炒蛋"等食谱。小陈选了一个收藏，顺便买了所需的其他食材。
- **核心功能描述**：
  1. 用户上传：拍摄或从相册选择食材照片（支持多图）
  2. AI 识别：识别图中食材名称、估算数量/重量、判断新鲜度（可选）
  3. 结果展示：识别出的食材列表（带置信度）、基于这些食材的推荐食谱、直接跳转 AI 食谱生成
  4. 交互优化：识别失败时允许用户手动纠正，纠正后的数据用于模型优化（未来）
- **技术依赖**：
  - 多模态 LLM API：GPT-4o / Gemini Pro Vision / 通义千问 VL / Claude 3 Sonnet
  - 图片上传与存储服务（复用现有文件存储）
  - 图像预处理（压缩、裁剪，降低 API 成本）
- **预估工作量**：2-3 周（1 名后端 + 1 名前端）
  - 图片上传与预处理：3 天
  - 多模态 API 接入与 Prompt 设计：5 天
  - 前端拍照/选图 UI：5 天
  - 识别结果展示与纠错交互：3 天
- **风险/注意事项**：
  - ⚠️ **成本问题**：多模态 API 比纯文本贵 5-10 倍。假设日活 1000，每人每天 1 次识别，月成本可能达到 ¥500-2000（取决于 API 选型）
  - ⚠️ 识别准确率：光线、角度、遮挡都会影响结果。相似食材（如韭菜 vs 蒜苗）容易混淆
  - ⚠️ 使用频率存疑：文字输入食材对多数用户已足够便捷，拍照识别的增量价值有限
  - ⚠️ 新鲜度判断不可靠：AI 对新鲜度的判断主观性强，不应作为主要卖点
  - ✅ 建议：作为差异化亮点功能推出，但控制免费使用次数（如每月 5 次免费，超出收费或看广告），并优先选择性价比高的国产多模态 API（通义千问 VL、文心一言等）

## 推荐实施路线

### 阶段一：P0 快速落地（第 1-3 周）
**目标**：上线 AI 食谱生成，验证用户接受度

1. **Week 1**：Prompt 工程 + 后端 API 原型
   - 设计结构化输出 Schema（食谱名称、食材、步骤、时间、难度）
   - 接入 DeepSeek / 通义千问 API（性价比高，中文表现好）
   - 完成生成 API 和保存到草稿箱功能
2. **Week 2**：前端开发
   - 设计生成表单（食材输入、偏好选择）
   - 食谱生成结果展示页（美观、可读性强）
   - 一键发布/收藏/重新生成交互
3. **Week 3**：测试优化 & 上线
   - 内部测试 50+ 条生成请求，调优 Prompt
   - 收集种子用户反馈
   - 灰度发布，监控生成质量和 API 成本

### 阶段二：P1 功能扩展（第 4-8 周）
**目标**：在食谱生成基础上，增加菜单规划和营养分析

4. **Week 4-5**：智能菜单规划
   - 复用 AI 食谱生成能力，构建周规划 Prompt
   - 开发日历视图和购物清单
   - 上线基础版，收集用户使用率数据
5. **Week 6-8**：营养分析
   - 先用 LLM 估算实现 MVP（快速上线，标注"仅供参考"）
   - 设计营养卡片 UI（热量、三大营养素、标签）
   - 后台跑批量分析，为存量食谱补全营养数据

### 阶段三：P2 差异化亮点（第 9-12 周，视资源而定）
**目标**：上线图片识别，形成差异化竞争力

6. **Week 9-10**：图片识别食材
   - 接入多模态 API（优先国产，控制成本）
   - 实现拍照 → 识别 → 推荐食谱的闭环
   - 设计免费额度 + 付费/广告解锁机制
7. **Week 11-12**：全功能优化 & 数据驱动迭代
   - 分析各功能的使用数据（DAU、留存、转化率）
   - 优化使用频率低的功能，或调整运营策略
   - 考虑接入专业营养数据库（如中国食物成分表），替换 LLM 估算

## 技术建议

### API 选型建议

| 功能 | 推荐方案 | 备选方案 | 选择理由 |
|------|---------|---------|---------|
| AI 食谱生成 | **DeepSeek-V3 / 通义千问-Plus** | GPT-4o / Claude 3.5 | 国产模型中文理解好，成本低 5-10 倍 |
| 智能菜单规划 | 复用上述 LLM，加系统 Prompt | 同上 | 无需额外模型 |
| 营养分析（MVP）| 复用上述 LLM | 同上 | 快速验证需求 |
| 营养分析（精准版）| **中国食物成分表 + LLM 解析** | USDA FoodData Central | 中式食材更精准 |
| 图片识别食材 | **通义千问-VL / 文心一言-VL** | GPT-4o Vision / Gemini Pro Vision | 国产多模态性价比高，支持中文场景 |

### 成本估算（月度，假设日活 1000）

| 功能 | 日调用量 | 单次成本 | 月度成本估算 |
|------|---------|---------|------------|
| AI 食谱生成 | 2000 次 | ¥0.005-0.01 | ¥300-600 |
| 智能菜单规划 | 500 次 | ¥0.01-0.02 | ¥150-300 |
| 营养分析 | 1000 次 | ¥0.002-0.005 | ¥60-150 |
| 图片识别食材 | 500 次 | ¥0.05-0.15 | ¥750-2250 |
| **合计** | | | **¥1260-3300** |

> 💡 **省钱策略**：
> 1. 使用 DeepSeek 或通义千问等国产模型，成本比 GPT-4 低 70-90%
> 2. 营养分析先用 LLM 估算，用户量大了再考虑自建数据库
> 3. 图片识别设置免费额度（每月 3-5 次），超出后引导付费或观看激励视频
> 4. 对热门食谱的营养分析结果做缓存，避免重复计算

### 技术架构建议

```
用户请求
  ├──→ AI 食谱生成 → LLM API → 结构化 JSON → 前端展示
  ├──→ 智能菜单规划 → LLM API（复用）→ 日历/清单 UI
  ├──→ 营养分析 → LLM 估算 / 数据库查询 → 营养卡片
  └──→ 图片识别食材 → 图像压缩 → 多模态 LLM → 食材列表
```

- **统一封装 LLM 调用层**：便于切换模型、监控用量、降级处理（某个模型挂了自动切另一个）
- **结果缓存**：相同食材组合的生成结果缓存 24h，降低成本
- **异步生成**：复杂请求（如 7 天菜单规划）可异步处理，前端轮询或 WebSocket 推送
- **限流与熔断**：防止用户恶意刷 API，设置单用户日调用上限

### 数据安全与合规

- 用户上传的食材图片可能包含敏感信息（如家庭环境），需明确告知用户图片仅用于识别
- AI 生成的食谱建议添加免责声明："AI 生成内容仅供参考，请根据实际情况调整，涉及饮食健康请咨询专业人士"
- 营养分析数据标注"估算值"，避免用户作为医疗依据
