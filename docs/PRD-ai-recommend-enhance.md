## 1. 用户痛点分析

### 1.1 当前推荐功能的不足

| # | 痛点 | 具体表现 | 严重度 |
|---|------|----------|--------|
| P1 | **AI 仅作为兜底，未被充分利用** | AI 仅在数据库无匹配结果时才调用，绝大多数场景用户只看到粗粒度的 LIKE 模糊匹配结果，AI 能力被浪费 | 🔴 高 |
| P2 | **匹配逻辑粗糙，缺少语义理解** | 输入"鸡胸肉"无法匹配到含"鸡肉"的食谱，"土豆"匹配不到"马铃薯"，缺少同义词/近义词扩展 | 🔴 高 |
| P3 | **推荐结果单一，缺少多样性** | 只按匹配度排序，用户连续搜索看到相似结果；无口味、菜系、难度维度的差异推荐 | 🟡 中 |
| P4 | **没有"缺什么"的提示** | 用户看到 60% 匹配度，不知道缺的 40% 是什么食材、能否用替代品，无从决策 | 🟡 中 |
| P5 | **AI 生成菜谱不可操作** | AI 推荐的菜谱只有标题/食材/简介，无详细步骤，用户看了没法做，体验断裂 | 🔴 高 |
| P6 | **没有个性化** | 新用户和老用户看到完全一样的推荐，不考虑历史行为、口味偏好、烹饪水平 | 🟡 中 |
| P7 | **单一轮交互** | 用户搜一次看到结果就结束，无法像和厨师聊天一样"换个口味""再简单点""人多一点怎么调" | 🟡 中 |
| P8 | **缺少场景化引导** | 用户不知道搜什么——"冰箱里剩了点东西想做顿饭"这种真实场景没有入口 | 🟡 中 |
| P9 | **搜索历史过于简单** | 只有文本列表，不能从历史中学习偏好，也无法快速复用上次搜索结果的优化 | 🟢 低 |

### 1.2 用户场景画像

| 画像 | 典型行为 | 当前痛点 |
|------|----------|----------|
| **冰箱清道夫** | 打开冰箱看到零散食材，想一次用完 | 需要手动输入所有食材，无"清空冰箱"引导 |
| **厨房新手** | 不知道手头食材能做什么 | 匹配度数字看不懂，缺食材不知怎么替代 |
| **时间紧张的上班族** | 30 分钟内做出一顿饭 | 无法按烹饪时间筛选推荐结果 |
| **营养关注者** | 想知道这顿饭营养够不够 | 完全没有营养信息 |
| **挑食/有忌口** | 不吃辣、对海鲜过敏 | 无法排除特定食材或口味 |

## 2. 功能增强方案

### 2.1 🧊 清空冰箱模式（Fridge Clear）

**简述**：一键输入冰箱剩余食材，AI 智能组合推荐"能消耗最多食材"的菜谱，优先推荐能用完全部食材的方案。

**用户价值**：
- 解决"冰箱里零零散散一堆东西怎么处理"的真实痛点
- 减少食材浪费，降低生活成本
- 游戏化体验：显示"清空率"让用户有成就感

**AI 如何发挥作用**：
1. AI 对用户输入的食材列表做**食材归一化**（土豆=马铃薯、鸡胸肉=鸡肉），扩展匹配范围
2. AI 生成"清空优先"排序策略：优先推荐食材利用率最高的菜谱
3. 当数据库菜谱无法消耗全部食材时，AI 生成**组合方案**（如"做 A 菜用掉鸡蛋番茄，做 B 菜用掉剩的豆腐"）
4. AI 对缺的少量辅助食材（如盐、酱油等调料）自动**豁免**，不降低匹配度

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | 新增 `FridgeMode` 入口（Hero 区 Tab 切换"精准推荐"/"清空冰箱"）；食材输入改为**标签式多选**（支持快速删除单个食材）；结果卡片增加"清空率"标签和进度条 |
| 后端 | 新增 `POST /api/recipes/fridge-clear` 接口；AI Prompt 增加清空优先指令；计算食材利用率排序逻辑 |
| 数据 | 无新增表，复用 Recipe 模型 |

### 2.2 🔄 AI 食材替代建议（Ingredient Substitute）

**简述**：当推荐食谱中缺少部分食材时，AI 自动建议可替代的食材，用户一键替换后重新计算匹配度。

**用户价值**：
- 消除"差一种食材就不做了"的障碍
- 拓展用户的烹饪知识和食材认知
- 大幅提升推荐结果的可用性（从"60% 匹配"变成"90% 可做"）

**AI 如何发挥作用**：
1. 对每条推荐结果中**未匹配的食材**，AI 生成替代建议列表（含替代理由）
2. 替代建议考虑：口味相似度、质地相似度、烹饪方式适配度
3. 替代建议标注置信度（高/中/低），高置信度替代可自动应用
4. 示例：食谱需要"蚝油"但用户没有 → AI 建议"生抽+少许糖可替代蚝油（鲜味相近）"

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | 推荐卡片中未匹配食材标签变为可点击，点击弹出替代建议气泡；用户选择替代后，匹配度实时重算并显示"✅ 已替代"标记 |
| 后端 | 新增 `POST /api/recipes/substitute` 接口；输入：目标食谱 ID + 缺少的食材列表；输出：替代建议数组；调用 AI 生成替代方案 |
| 数据 | 无新增表 |

### 2.3 💬 AI 对话式推荐（Conversational Recommend）

**简述**：在推荐结果下方增加对话面板，用户可以用自然语言 refine 推荐，如"换个辣的""再简单点""不想吃猪肉""多推荐点汤"。

**用户价值**：
- 从"一锤子买卖"变成"像和厨师聊天"，大幅提升推荐体验
- 降低用户重新搜索的成本（不用清空重新输入）
- 每轮对话的上下文被记住，推荐越来越精准

**AI 如何发挥作用**：
1. 后端维护一个**对话上下文**（sessionId + 历史消息），每轮对话将上下文传给 AI
2. AI 返回结构化的调整指令（筛选条件变更）+ 自然语言回复
3. 支持 7 种常见调整意图识别：
   - 口味调整："换个辣的" → flavor 变为"麻辣"
   - 难度调整："再简单点" → difficulty 变为"easy"
   - 食材排除："不想吃猪肉" → excludeIngredients 增加"猪肉"
   - 菜系调整："来点西餐" → cuisine 变为"西餐"
   - 时间调整："30分钟能搞定的" → maxCookTime = 30
   - 人数调整："4 个人吃" → servings = 4
   - 换一批："看看别的" → offset += pageSize
4. 对话历史存储在服务端（Redis/内存），设置 30 分钟过期

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | 推荐结果下方新增对话面板组件（ChatPanel）；支持快捷回复标签 + 自由文本输入；对话气泡 UI；对话中的筛选条件实时反映到结果列表 |
| 后端 | 新增 `POST /api/recipes/chat` 接口（流式 SSE 返回）；新增 session 管理（内存 Map，key 为 sessionId）；AI Prompt 包含对话历史 + 当前筛选状态 |
| 数据 | 可选新增 `ChatSession` 表（用于持久化，V2） |

### 2.4 📊 AI 营养分析与分量调整（Nutrition & Servings）

**简述**：为推荐食谱提供 AI 生成的营养概览（热量/蛋白质/碳水/脂肪），并支持根据用餐人数智能调整食材分量。

**用户价值**：
- 健康饮食越来越受关注，营养信息是决策关键因素
- 分量调整解决"食谱写 2 人份但我 4 个人吃"的普遍问题
- 营养标签提升网站专业度和信任感

**AI 如何发挥作用**：
1. **营养分析**：将食谱食材列表发给 AI，返回每份营养估算：
   ```
   { calories: 450, protein: 25g, carbs: 30g, fat: 20g, fiber: 5g }
   ```
2. **分量调整**：用户调整人数后，AI 按比例缩放食材用量，同时处理非线性缩放（如盐不需要 2x、某些食材缩放有上限）
3. **膳食均衡提示**：如果用户连续搜索，AI 可提示"你今天的推荐偏碳水，建议搭配一道蔬菜"

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | 推荐卡片增加营养标签条（热量+三大营养素）；食谱详情页增加"👥 人数调整"滑块/按钮；人数变化时食材用量实时更新；新增营养详情弹窗 |
| 后端 | 新增 `POST /api/recipes/nutrition` 接口（批量营养分析）；新增 `POST /api/recipes/adjust-servings` 接口（分量调整）；两者均调用 AI |
| 数据 | Recipe 模型可选新增 `nutrition` 字段（缓存 AI 结果，避免重复调用） |

### 2.5 🌡️ 应季食材推荐（Seasonal Recommend）

**简述**：根据当前月份自动推荐应季食材和对应食谱，降低用户"不知道搜什么"的决策成本。

**用户价值**：
- 应季食材更新鲜、更便宜、更健康，但用户往往不知道什么季节吃什么
- 解决"不知道搜什么"的冷启动问题
- 增加内容发现感，让推荐页面本身就有浏览价值

**AI 如何发挥作用**：
1. AI 根据当前月份生成**应季食材清单**（考虑地域：默认中国/可配置）
2. AI 为每种应季食材匹配数据库中最合适的食谱
3. 生成应季推荐理由（如"5 月的蚕豆正嫩，错过再等一年"）
4. 可扩展：结合用户地理位置（未来）

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | Hero 区新增"🌡️ 本月时令"Tab/模块；展示应季食材卡片（带推荐理由）；点击食材直接触发搜索 |
| 后端 | 新增 `GET /api/recipes/seasonal?month=5` 接口；调用 AI 生成应季食材 + 推荐食谱；结果可缓存（按月缓存） |
| 数据 | 可选新增 `seasonal_cache` 表（按月缓存 AI 结果，减少 API 调用） |

### 2.6 🧠 AI 智能排序 + 个性化（Smart Ranking）

**简述**：超越简单的匹配度排序，综合用户偏好、烹饪难度、口味多样性等维度智能排序推荐结果。

**用户价值**：
- 每个人看到的推荐结果不一样，"为我推荐"而非"按规则排序"
- 新手看到简单的、老饕看到有挑战的
- 避免"10 条结果 8 条都是炒菜"的同质化

**AI 如何发挥作用**：
1. **用户偏好建模**：基于用户收藏/浏览历史，AI 总结口味画像（偏辣/偏清淡/偏好菜系等）
2. **多维度排序**：AI 对每条结果打分，综合考虑：
   - 食材匹配度（40% 权重）
   - 口味偏好匹配（25%）
   - 烹饪难度匹配（15%，新手偏好简单）
   - 多样性去重（10%，避免同菜系扎堆）
   - 时令加成（10%，应季食材加分）
3. **"为什么推荐给你"**：AI 为每条结果生成一句话推荐理由

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | 推荐卡片增加"为你推荐"标签 + AI 推荐理由；排序方式切换（智能/匹配度/时间/难度） |
| 后端 | `GET /api/recipes/recommend` 增加 `sortStrategy` 参数；新增 `POST /api/user/preference` 接口（更新偏好）；推荐接口集成 AI 排序逻辑 |
| 数据 | 可选新增 `UserPreference` 表（存储用户口味画像）；Recipe 增加向量嵌入字段（未来语义搜索基础） |

### 2.7 🤖 AI 菜谱生成增强（Enhanced AI Recipe Generation）

**简述**：当 AI 生成菜谱时，补充完整的烹饪步骤、小贴士和营养信息，使 AI 菜谱与数据库菜谱体验一致。

**用户价值**：
- 当前 AI 菜谱只有标题和食材，用户没法照着做——体验断裂
- 补全步骤后，AI 生成的菜谱可以直接用于烹饪
- 消除"数据库食谱"和"AI 食谱"的体验鸿沟

**AI 如何发挥作用**：
1. 优化 AI Prompt，要求返回完整结构：
   ```json
   {
     "title": "番茄鸡蛋盖饭",
     "description": "...",
     "ingredients": [...],
     "steps": [
       { "stepNumber": 1, "content": "番茄切块，鸡蛋打散...", "tip": "鸡蛋加少许水会更嫩" },
       ...
     ],
     "cookTime": 15,
     "difficulty": "easy",
     "servings": 2,
     "nutrition": { "calories": 380, "protein": 18, "carbs": 35, "fat": 15 },
     "tips": "番茄先划十字用开水烫更容易去皮"
   }
   ```
2. AI 生成步骤需满足：每个步骤可操作、有具体时间/火候描述
3. 增加"烹饪小贴士"字段，提供实用技巧
4. **一键保存**：用户可将 AI 生成的菜谱保存为自己的食谱

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | AI 菜谱卡片样式对齐数据库食谱；增加"展开步骤"折叠区；增加"保存到我的食谱"按钮；步骤展示组件复用 RecipeDetailPage |
| 后端 | 修改现有 recommend 接口的 AI Prompt；新增 `POST /api/recipes/save-ai` 接口（将 AI 生成菜谱持久化为正式食谱） |
| 数据 | 无新增表，复用 Recipe 模型（增加 `source: 'ai'` 字段可选） |

### 2.8 🚫 食材排除与饮食偏好（Dietary Filter）

**简述**：支持用户设置饮食偏好（忌口、过敏、素食等）和排除食材，推荐结果自动过滤。

**用户价值**：
- 对海鲜过敏、素食者、不吃辣的用户，当前推荐完全无法满足
- 一次设置、持续生效，无需每次搜索都手动排除

**AI 如何发挥作用**：
1. 用户设置饮食标签后，AI 在推荐时**自动排除**不合规食谱
2. AI 识别食谱中的隐含食材（如"蚝油"含海鲜成分，"豆瓣酱"含大豆）
3. 当排除后结果不足时，AI 主动推荐替代方案

**前后端改动范围**：

| 层 | 改动 |
|----|------|
| 前端 | 设置页新增"饮食偏好"模块；预设标签：素食/清真/无辣/无海鲜/无坚果/低碳水等；搜索框旁增加"排除食材"快捷入口 |
| 后端 | 新增 `GET/PUT /api/user/dietary` 接口；推荐接口增加 `exclude` 和 `dietary` 过滤参数 |
| 数据 | 新增 `UserDietary` 表（userId + dietaryTags + excludeIngredients） |

## 3. 优先级分级

| 优先级 | 功能 | 理由 |
|--------|------|------|
| **P0** | 2.7 AI 菜谱生成增强 | 当前 AI 推荐只有标题+食材无步骤，是最大体验断裂点，修复成本最低、收益最高 |
| **P0** | 2.2 AI 食材替代建议 | 核心差异化功能，解决"差一种食材就不做了"的普遍痛点，AI 价值发挥最直接 |
| **P1** | 2.1 清空冰箱模式 | 最具传播力的场景，用户真实需求强烈，且有社交传播潜力 |
| **P1** | 2.3 AI 对话式推荐 | 体验升级最显著，但实现复杂度较高，先做基础版 |
| **P1** | 2.8 食材排除与饮食偏好 | 基础过滤能力，没有它很多用户无法使用推荐功能 |
| **P2** | 2.4 营养分析与分量调整 | 价值明确但非刚需，适合作为差异化亮点 |
| **P2** | 2.5 应季食材推荐 | 增加浏览量和停留时间，但不影响核心推荐体验 |
| **P2** | 2.6 AI 智能排序+个性化 | 需要用户行为数据积累，前期数据不足效果有限 |

### 优先级决策逻辑图

```
P0: 体验断裂 → 必须修复
  └─ AI菜谱无步骤 → 用户无法使用（2.7）
  └─ 匹配度低无可操作建议 → 推荐无意义（2.2）

P1: 核心体验升级
  └─ 场景化入口 → 降低使用门槛（2.1）
  └─ 交互升级 → 提升推荐效率（2.3）
  └─ 基础过滤 → 扩大可用人群（2.8）

P2: 差异化亮点
  └─ 营养信息 → 专业度提升（2.4）
  └─ 应季引导 → 内容发现（2.5）
  └─ 个性化排序 → 长期价值（2.6）
```

## 4. 交互流程

### 4.1 清空冰箱模式（Fridge Clear）

```
用户进入 /recommend
  └─ 点击"🧊 清空冰箱"Tab
```

## 6. 技术实现备注

### 6.1 AI 调用策略

| 策略 | 说明 |
|------|------|
| **超时控制** | 所有 AI 调用设置 15s 超时，超时后降级到纯数据库结果 |
| **重试** | AI 调用失败最多重试 1 次，间隔 1s |
| **并发控制** | 单次请求最多发起 2 个并行 AI 调用（如：匹配 + 替代建议） |
| **缓存** | 应季推荐按月缓存；营养分析按食谱缓存；对话不缓存 |
| **Token 控制** | 单次 AI 请求 max_tokens ≤ 2000，避免响应过长 |
| **流式返回** | 对话式推荐使用 SSE 流式返回，提升体验 |

### 6.2 后端新增接口汇总

## 附录 A：AI 特色场景评估

对需求中列出的 7 个 AI 特色场景的评估结论：

| 场景 | 是否值得做 | 优先级 | 理由 |
|------|-----------|--------|------|
| 清空冰箱菜谱 | ✅ 强烈推荐 | P1 | 最强场景入口，用户需求真实，传播性好 |
| 应季食材推荐 | ✅ 推荐 | P2 | 增加浏览深度，降低冷启动门槛，但非刚需 |
| 营养分析 | ✅ 推荐 | P2 | 差异化亮点，健康趋势下价值递增 |
| 食材替代 | ✅ 强烈推荐 | P0 | AI 价值最直接的体现，解决核心痛点 |
| 智能排序 | ⚠️ 条件推荐 | P2 | 需用户行为数据积累，前期效果有限；但长期必做 |
| 对话式推荐 | ✅ 推荐 | P1 | 体验升级最显著，但实现复杂度高，先做基础版 |
| 分量调整 | ✅ 推荐 | P2 | 实用且低复杂度，与营养分析天然搭配 |

**关键决策**：
- **食材替代**是最小投入最大产出的功能，必须优先做
- **清空冰箱**是最有传播力的场景，值得投入资源
- **对话式推荐**是体验升级的核心，但需要分阶段实现（先意图识别 → 再自由对话）
- **智能排序**需要数据积累，初期用规则 + AI 混合策略，后期逐步强化 AI 权重
