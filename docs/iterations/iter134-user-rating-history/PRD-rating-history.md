## 1. 背景与目标

### 1.1 背景
迭代 #130 + #132 + #133 已上线 4 维评分系统（taste / difficulty / presentation / value），目前用户只能在单个食谱详情页看到该食谱的 4 维雷达图聚合数据。

现有数据规模：696+ 条评论，其中 92.7% 含 4 维评分。

### 1.2 问题
用户无法在个人维度回顾自己的评分历史：
- 没有"我整体的口味偏好是什么"的答案
- 没有"我最近评分趋势是更挑剔还是更宽松"的轨迹
- 不知道自己在 4 个维度上的分布特征（是给分集中在 4-5，还是 1-3 都给）
- 想找回"之前给过高分/低分的某道菜"很困难

### 1.3 目标
在 `UserProfilePage`（`/profile/{userId}`）新增"评分历史"模块，让用户能：
1. **回看**自己所有评分过的食谱及对应分数
2. **理解**自己在 4 个维度上的偏好画像
3. **发现**自己评分行为的变化趋势
4. **定位**自己曾经给过高分/低分的关键食谱

### 1.4 非目标（MVP 阶段不做）
- 推荐"你应该再评一下这些菜"
- 评分预测（"这道菜你可能给 4.5 分"）
- 跨用户对比（"你和美食家 #42 的口味相似度 87%"）
- 评分时间提醒
- 评分导出 / 数据下载
- 公开"评分历史"分享卡片（截图分享到外部）

## 2. 用户故事

### US-1：核心场景 - 回顾自己的评分
**As a** 登录用户
**I want** 在个人主页看到自己所有评分过的食谱列表
**So that** 我能快速回忆"我曾经对哪些菜给出过评价"，并能点击回到具体食谱

### US-2：口味画像
**As a** 一个有 10+ 条评分的活跃用户
**I want** 看到自己 4 个维度的平均分雷达图（taste / difficulty / presentation / value）
**So that** 我能直观了解"我是味觉敏感型"还是"我是颜值党"还是"我更看重性价比"

### US-3：与全站平均的对比
**As a** 一个好奇自己评分是否"严格"的用户
**I want** 在 4 维平均分旁边看到与全站平均分的差值（+0.3 / -0.2）
**So that** 我知道自己相对整个社区是更挑剔还是更宽松

### US-4：评分趋势
**As a** 一个有 20+ 条评分的老用户
**I want** 看到一条按时间排列的折线图，展示 4 个维度评分如何随时间变化
**So that** 我能发现"我最近几个月对味道的要求是不是变高了"或"我对难度的容忍度在上升"

### US-5：评分分布
**As a** 一个想了解自己评分习惯的用户
**I want** 看到 4 个维度各自的频次分布直方图（1-5 分我各给了多少次）
**So that** 我知道自己是不是"5 分党"或"3 分中庸党"

### US-6：高/低分 TOP 列表
**As a** 一个想重温自己"心头好"和"踩雷"的用户
**I want** 看到"我给分最高的 5 道菜"和"我给分最低的 5 道菜"两个列表
**So that** 我能快速回到那些让我印象深刻的食谱

### US-7：少数据状态
**As a** 只评过 1-2 条评论的新用户
**I want** 看到友好的引导而不是空表格
**So that** 我知道"再多评几条就能解锁完整画像"

### US-8：隐私意识
**As a** 注重隐私的用户
**I want** 能选择是否公开我的评分历史
**So that** 我既可以选择分享给社区，也可以保留私人空间

## 4. 业务规则

### 4.1 可见性矩阵

| 角色 | 查看他人评分历史 | 查看自己评分历史 |
|---|---|---|
| 未登录用户 | ❌ 看不到（看到的是引导登录） | N/A |
| 登录用户，隐私 = 公开 | ✅ 看到完整模块 | ✅ 看到完整模块 |
| 登录用户，隐私 = 私有 | ❌ 看到"该用户未公开评分历史" | ✅ 看到完整模块 |
| 自己的主页（任何状态） | N/A | ✅ 永远完整可见（不受隐私设置影响） |

### 4.2 数据筛选规则

| 规则 | 处理方式 |
|---|---|
| 匿名评论（无 userId） | **不计入**个人评分历史，无法追踪 |
| 4 维全部为 0 的旧评论 | 从 4 维统计中过滤掉，**但**仍计入"总评论数"和"总体平均分" |
| 单维度为 0 但其他有效 | 该维度视为"用户未评分"，**不参与该维度平均**；但其他 3 个维度正常参与 |
| 已删除的评论 | 立即从历史中消失，缓存同步失效 |
| 草稿状态评分 | 不计入历史（如果未来引入评分草稿功能） |
| 同一食谱多次评分 | 按**最新一次**评分计入历史和统计（旧版本不展示） |

### 4.3 隐私默认值

- 新用户默认 = **公开**（与现有个人主页公开一致，降低首次体验阻力）
- 用户可随时在「设置 → 隐私 → 评分历史可见性」切换
- 切换为"私有"后：他人访问用户主页时该模块替换为提示；缓存和搜索引擎索引在 24 小时内清理
- 公开 = 任何人可看完整模块；不存在"仅好友可见"中间态（避免复杂度爆炸）

### 4.4 空状态分级

| 状态 | 触发条件 | 展示策略 |
|---|---|---|
| 全空 | 0 条带 userId 的评论 | 完整引导插画 + "你还没有评过任何菜，去看看？" + CTA 跳首页 |
| 极少 | 1-2 条评分 | 4 维平均分显示具体值（带 ⚠️ 提示）；雷达图仍显示但下方有"样本太少，建议多评几条"；趋势图**不显示**，替换为"再评 X 条就能看到趋势"；分布图**不显示** |
| 少 | 3-4 条评分 | 同上但所有图表区域可显示，无"不显示"模块；顶部增加"样本较少"提示条 |
| 充足 | 5-9 条评分 | 全部模块正常显示，分布图保留"样本较少"角标 |
| 丰富 | 10+ 条评分 | 完整模块，无任何"样本不足"提示 |

## 5. 验收标准

### 5.1 数据正确性

**AC-01**：4 维平均分计算时，**仅**纳入该维度大于 0 的评分；用 SQL `AVG(tasteRating) WHERE tasteRating > 0` 验证，结果保留 1 位小数。

**AC-02**：高/低分 TOP 5 中"高"按 `overall` 降序取前 5，"低"按 `overall` 升序取前 5；同一食谱多次评分时只取最新一次。

**AC-03**：删除一条评论后，**30 秒内**该评分从历史列表、平均分、分布、趋势、T5/L5 全部消失（缓存 TTL 内有效，但前端拉取新数据后立即生效）。

### 5.2 业务规则

**AC-04**：未登录用户访问 `/profile/{anyUserId}` 看到"该用户的评分历史"模块区域被替换为"登录后查看完整评分历史"引导。

**AC-05**：隐私设置为"私有"的用户，他人访问其 `/profile/{userId}` 时评分历史模块区域显示"该用户未公开评分历史"；本人访问自己时仍可见完整。

**AC-06**：评分全部维度为 0 的旧评论**不**出现在 4 维平均分计算中，但仍计入该用户的"总评分次数"徽章（如有）。

### 5.3 交互完整性

**AC-07**：历史列表的"查看更多"按钮每次精确加载 10 条，加载中显示 spinner，加载完按钮消失或禁用；总条数 = 0 时按钮不显示。

**AC-08**：历史列表点击任意一行，路由跳转到该食谱详情页（`/recipes/{recipeId}`），URL 携带 `from=profile` 参数便于回溯。

**AC-09**：4 维雷达图 hover 任一顶点，显示该维度的具体平均分 + "基于 X 次评分"。

### 5.4 性能

**AC-10**：单次首屏加载（包含 4 维平均 + 雷达图 + 趋势 + 分布 + TOP 5 + 最近 10 条历史）**首字节 < 500ms**，**完整渲染 < 1.5s**（P95 在 4 核 8G 普通服务器下）。加载策略可并发，但首屏数据接口总响应 < 800ms（P95）。

## 6. 数据需求

### 6.1 后端 API 设计

#### API-1：用户评分聚合统计
```
GET /api/users/:userId/ratings/summary
```

**用途**：F-01, F-02, F-04, F-05, F-06 复用

**Query 参数**：
- `timeRange`：`all` | `30d` | `90d` | `1y`（可选，默认 `all`）

**Response 结构**（示意）：
```json
{
  "userId": 123,
  "totalRatings": 47,
  "validDimensionRatings": 43,
  "dimensionAverages": {
    "taste":        { "average": 4.2, "count": 43, "siteAverage": 3.8, "delta": 0.4 },
    "difficulty":   { "average": 3.1, "count": 42, "siteAverage": 3.4, "delta": -0.3 },
    "presentation": { "average": 4.5, "count": 40, "siteAverage": 3.9, "delta": 0.6 },
    "value":        { "average": 3.8, "count": 43, "siteAverage": 3.7, "delta": 0.1 }
  },
  "distribution": {
    "taste":        { "1": 0, "2": 1, "3": 5, "4": 18, "5": 19 },
    "difficulty":   { "1": 2, "2": 8, "3": 20, "4": 10, "5": 2 },
    "presentation": { "1": 0, "2": 0, "3": 4, "4": 14, "5": 22 },
    "value":        { "1": 1, "2": 3, "3": 12, "4": 20, "5": 7 }
  },
  "trend": {
    "interval": "month",
    "points": [
      { "period": "2025-12", "taste": 4.1, "difficulty": 3.0, "presentation": 4.4, "value": 3.7, "count": 5 },
      { "period": "2026-01", "taste": 4.3, "difficulty": 3.2, "presentation": 4.5, "value": 3.9, "count": 8 }
    ]
  }
}
```

**注意**：
- `siteAverage` 来自全站聚合缓存（可走 `GET /api/recipes/comments/global-stats` 已有端点，或本接口后端内联）
- `count < 3` 的 `period` 仍返回（前端决定是否展示）
- `distribution` 的 key 是字符串化的分值（1-5），value 是该分值的次数

#### API-2：评分历史列表
```
GET /api/users/:userId/ratings/history
```

**用途**：F-03, F-07, F-09, F-10

**Query 参数**：
- `page`：页码（默认 1）
- `pageSize`：每页条数（默认 10，最大 20）
- `sort`：`time_desc`（默认） | `rating_desc` | `rating_asc`
- `timeRange`：可选
- `dimension`：可选（`overall` | `taste` | `difficulty` | `presentation` | `value`，仅在 `sort` 为 `rating_*` 时生效）

**Response 结构**（示意）：
```json
{
  "userId": 123,
  "total": 47,
  "page": 1,
  "pageSize": 10,
  "items": [
    {
      "commentId": 8888,
      "recipeId": 456,
      "recipeTitle": "麻婆豆腐（家常版）",
      "recipeCoverUrl": "https://cdn.../xxx.jpg",
      "ratings": {
        "taste": 5,
        "difficulty": 2,
        "presentation": 4,
        "value": 5,
        "overall": 4
      },
      "commentText": "花椒香...",
      "createdAt": "2026-05-21T10:23:00+08:00"
    }
  ]
}
```

#### API-3：隐私设置
```
GET  /api/users/:userId/privacy
PUT  /api/users/:userId/privacy
```

**Body（PUT）**：
```json
{
  "ratingsHistoryPublic": true
}
```

**Response**：
```json

```

#### API-4：TOP 5 列表
```
GET /api/users/:userId/ratings/top?type=high|low&limit=5
```

**用途**：F-07

**Response**：结构同 API-2 `items` 字段，但只含 5 条。

#### 是否需要新增 API 的判断
- F-01~F-02, F-04~F-06：1 个聚合 API 即可
- F-03, F-09~F-10：1 个分页历史 API
- F-07：可以复用 history API + 排序参数，但**建议**独立端点（数据量小、可缓存、查询逻辑更简单）
- F-11：复用现有 user settings 端点（如果存在），否则新增

### 6.2 数据来源

后端从 `Comments` 表查询（已有字段）：
- `userId`（外键）
- `recipeId`（外键）
- `tasteRating`, `difficultyRating`, `presentationRating`, `valueRating`（int 1-5，0 表示未评分）
- `rating`（overall 1-5）
- `isAnonymous`（如有）
- `createdAt`, `updatedAt`, `deletedAt`（如启用软删除）

需要 JOIN：
- `Recipes` 表（拿到标题、封面 URL、slug）
- `Users` 表（拿到 username 用于反查）

### 6.3 性能要求

| 项 | 目标 | 策略 |
|---|---|---|
| 聚合统计响应 | P95 < 200ms | 命中 Redis 缓存（TTL 5min）；缓存 key 含 userId + timeRange |
| 趋势图聚合 | P95 < 300ms | 后端预计算"月聚合"物化视图，每 10 分钟刷新一次 |
| 分布直方图 | P95 < 100ms | 4 个维度的 COUNT GROUP BY，索引 `userId + createdAt` |
| 历史分页 | P95 < 150ms | 索引 `userId + createdAt`（默认排序）或 `userId + rating`（按分排序） |
| TOP 5 | P95 < 100ms | 走索引 + LIMIT 5 |
| 首屏总耗时 | P95 < 800ms | summary + top + first page 三个请求并发 |
| 缓存失效 | 评论变更后 30s 内 | 监听 Comment 写操作，清除对应 userId 的缓存 key |
| 大数据量用户 | 1000+ 评分仍可秒级 | 分布/趋势按时间窗分桶后聚合，限制单次返回点数 ≤ 60（5 年 × 12 月） |

### 6.4 缓存策略

| 数据 | 缓存 Key | TTL | 失效触发 |
|---|---|---|---|
| summary | `rating:summary:{userId}:{timeRange}` | 5 分钟 | 任意该用户评分写入/删除 |
| top 5 | `rating:top:{userId}:{type}` | 5 分钟 | 同上 |
| trend | `rating:trend:{userId}:{timeRange}` | 10 分钟 | 同上（可放宽） |
| history 分页 | 不缓存 | - | - |
| 全站平均 | `site:dimAverages` | 1 小时 | 后台批处理每日凌晨刷新 |

## 7. UI / UX 方向

### 7.1 信息架构
模块在 `UserProfilePage` 中的位置：
- **第一优先级**（在 `StatsCharts` 上方，**`AchievementsPanel` 下方**），与现有"成就 / 统计 / 活动"并列
- 移动端折叠到 `StatsCharts` 之后

### 7.2 视觉调性
- **主色调**：沿用站点主色（暖橙/陶土色系），4 个维度用站点已有的固定色卡（与 `DimensionRadar` 一致：taste=橙 / difficulty=绿 / presentation=蓝 / value=黄）
- **delta 配色**：与全站平均正差 = 主色（积极），负差 = 中性灰（不强调"差"避免攻击性）
- **数据密度**：中等，避免单屏塞 20 个数字；优先用图形传达，数字作为辅助

### 7.3 关键交互
1. **历史列表**：每行 hover 抬起 + 阴影增强；点击整行跳转（不仅是标题）
2. **雷达图**：hover 顶点 → 浮窗显示数值 + 评分数；点击维度名 → 该维度线隐藏/显示（**P1**）
3. **趋势图**：hover 折线 → 浮窗显示该月 4 维数值 + 评分数；时间范围切换器在右上角（**P1**）
4. **分布直方图**：hover 柱子 → 显示"X 次给了 N 分"；长条悬停时主色加深
5. **TOP 5 切换**：用 Tab 切换"高分榜 / 低分榜"，避免两个列表同时占空间
6. **空状态**：使用插画 + 一句话引导 + 1 个 CTA 按钮；不要堆砌文字

### 7.4 动效趋势
- 模块首次进入视口：图表从骨架屏淡入真实数据（300ms）
- 数字（平均分、count）用 count-up 动画（500ms）增强"数据汇聚"感
- 列表"查看更多"加载：新 10 条从下方滑入（200ms stagger）
- **避免**：粒子效果、3D 旋转、复杂缓动；保持专业冷静

### 7.5 可访问性
- 所有图表提供 aria-label 描述（如"雷达图，4 维度口味画像，taste 4.2，difficulty 3.1"）
- 颜色不是唯一信息载体（图标/标签同时呈现）
- 键盘 Tab 可遍历列表项，Enter 触发跳转
- 暗色模式下颜色对比度 ≥ 4.5:1

### 7.6 响应式
- **桌面端（≥ 1024px）**：4 维平均分卡片 4 列横排；雷达图 + 趋势图左右分栏；分布 2×2 网格
- **平板（768-1023px）**：4 维平均分卡片 2×2 网格；图表单列堆叠
- **移动端（< 768px）**：4 维卡片单列；雷达图和趋势图单列；分布横向滚动

## 8. 风险与依赖

### 8.1 风险

| 风险 | 等级 | 缓解措施 |
|---|---|---|
| 7.3% 评论无 4 维评分被错误计入 | 中 | 严格按 4.2 规则过滤，AC-01 验证 |
| 大数据量用户（1000+ 评分）卡顿 | 中 | 6.3 节已规定聚合上限 + 缓存 |
| 隐私切换有缓存延迟导致短期数据泄露 | 低 | 切换时主动 invalidate 缓存（不依赖 TTL） |
| 与现有 `StatsCharts` 模块功能重复 | 中 | 明确分工：`StatsCharts` = 全站数据；本模块 = 个人数据；不重叠 |
| 雷达图视觉与单食谱页雷同导致用户混淆 | 低 | 标题明确"我的口味画像"vs 单食谱"XX 菜的口味评分" |

### 8.2 依赖
- **后端**：Comment 模型字段已就绪（tasteRating/difficultyRating/presentationRating/valueRating + overall）
- **后端**：需要新增 3 个聚合 API（见 6.1）
- **后端**：Comment 写操作需要加缓存失效 hook
- **前端**：复用 `DimensionRadar` 视觉规范和 `DashboardPage` 的 Recharts 组件库
- **前端**：新增隐私设置入口（可放在「设置」页或本模块右上角）
- **数据**：需要历史评论的全站聚合 baseline（用于 `siteAverage` 对比）

### 8.3 排期建议
- Day 1-2：后端 3 个 API + 缓存
- Day 3：后端联调 + 边界场景测试
- Day 4-5：前端 UI 实现（含 4 个图表）
- Day 6：空状态、隐私设置、响应式适配
- Day 7：联调 + 验收标准 10 条全过
